# 本地交付与支付方案分析

> 日期：2026-06-24
> 上下文：云端部署前端页面遇到障碍 → 重新评估不上云端的交付和收费方式
> 结论：推荐 Gumroad/Lemon Squeezy License Key 方案

---

## 一、现状

我们建了 HTML 用户系统（server.py 850行 + user.html SPA）用于云端注册和支付。
部署到阿里云后 `server.py` 和订阅 API 运行正常，但前端路由未触发导致页面白屏。

核心问题被重新审视：**如果用户装的是一个本地 Obsidian 插件，我们是否需要自建云端用户系统来收月费？**

---

## 二、核心挑战

CC 插件是一个本地 `.js` 文件——没有 DRM、可被无限复制。
收费需要解决的不是"怎么收钱"，而是"怎么让付了钱的用户能用，没付钱的不能用"。

---

## 三、六种方案对比

| 方案 | 怎么做 | 用户操作 | 开发量 | 收月费 |
|------|------|------|:--:|:--:|
| **A. 自建云端授权** | 插件调 `/api/verify-subscription` → 验证JWT | 网页注册→支付→自动 | ⚠️ 前端有bug | ✅ |
| **B. 本地 License Key** | 网站买Key→粘贴到插件→本地验证 | 网页买Key→复制→粘贴 | 🟢 低 | ❌ 需要支付渠道 |
| **C. Gumroad / Lemon Squeezy** | Gumroad 设订阅→自动发 Key→插件验证 | 点链接→付款→收Key→粘贴 | 🟢 极低 | ✅ 自动扣·平台代收 |
| **D. Obsidian 商店免费+付费激活** | 商店版功能受限→付费解锁 | 装插件→付费→输Key | 🟡 中 | ✅ 第三方 |
| **E. 按次计费** | 预充值·每次深挖扣 Token | 预付→用完再充 | 🔴 高 | ✅ |
| **F. 纯赞助** | 插件免费·底部放赞助链接 | 自愿 | 🟢 极低 | ❌ 不稳定 |

---

## 四、推荐方案：Gumroad / Lemon Squeezy（方案 C）

### 为什么

Gumroad 是专为独立开发者设计的数字产品销售平台。你只需：

1. 注册 Gumroad 账号 → 创建产品 → 设订阅价（¥48/月、¥488/年）
2. 配置自动续费
3. 平台自动处理：展示购买页、收款（微信/支付宝/Visa/Mastercard）、生成 License Key、每月自动扣款、取消退订
4. 插件端只需调 `https://api.gumroad.com/v2/licenses/verify` 验证 Key 是否有效

### vs 自建系统

| | 自建 HTML 系统 | Gumroad |
|------|:--:|:--:|
| 开发时间 | 4天 | **2小时** |
| 维护 | 持续 | **零** |
| 支付方式 | 微信/支付宝（需对接） | 全球·含微信支付宝 |
| 发票 | 自己做 | **自动** |
| 退款 | 自己做 | **后台一键** |
| 手续费 | 0.6% | **5-10%**（含支付费） |
| 用户体验 | 注册→登录→支付 | 点链接→付款→收Key→粘贴 |

平台费 5-10% 换来了零运维、零开发、全球支付支持。

## 五、如果选方案 C · 插件改动

```
插件设置面板新增：License Key 输入框
  ↓
用户粘贴 Gumroad 发的 Key → 点"验证"
  ↓
调 https://api.gumroad.com/v2/licenses/verify
  → 有效 → 存本地 → 正常使用
  → 无效 → 提示"无效"
  ↓
启动验证：
  → 本地检查 Key 哈希（离线可用）
  → 每 7 天调一次 Gumroad API 确认订阅有效
  → 取消 → 弹窗 → 禁用
```

**改动量**：ChatView.ts ~40 行。现有 `verifyLicense()` 可改造复用。
