# CLAUDE.md 重写经验总结

> 日期：2026-06-22 · 触发：OB+CC插件回复格式问题根治
> 改前：827行·回复=交付物清单 | 改后：171行·回复=自然投研对话

---

## 一、问题根因

### 信号比倒挂

```
旧 CLAUDE.md：23行"你是谁" vs 600+行"写什么卡" → 信号比 1:30
```

CC读到的是"你的任务是产出卡片和报告"。虽然§0号称"最高优先级"，但23行 vs 600行的体量差距让CC自然倾向于后者。

### 三个恶性循环

| # | 机制 | 效果 |
|---|------|------|
| 1 | 规则密度倒挂 | 600行写卡规则 → CC认为产出=卡片+报告 → 回复变成"交付物清单" |
| 2 | 对话历史自强化 | 坏格式一旦出现→存入stdin历史→下一轮CC读到→完美复制 |
| 3 | 任务过载 | 一次深挖CC同时做：财务分析+框架判断+写报告+写6张卡+交叉关联+MOC更新 |

## 二、为什么之前的14次修改全部无效

```
--system-prompt 注入  → 跟 CLAUDE.md 竞争，不稳定
§0 身份声明          → 被后面 600 行淹没
末尾加锚点            → 被 §11 卡片规则对冲
stdin 加行为提示      → 信号太弱
否定改成正面          → 底层问题不变
```

**根本矛盾**：CC 在 `-p` 模式下没有"聊天窗口"和"文件系统"的自然区分。不是 prompt 写得不够好——是信号强度差了 30 倍。

| 6 | 3个正面示例直接放在§0——格力/储能/依据不足 | CC模仿正确格式，而非被"不要"约束 |

## 六、卡片自动检测修复（ChatView.ts）

### 问题发现

CLA.md 重写后 CC 回复自然了，但深挖完成后插件提示"💡 本次未生成新卡片。说'生成卡片'可手动保存"——实际 CC 已经写了卡片。

### 根因：时间窗口检测的天生缺陷

旧版 `verifyCardsWritten()` 用 180 秒（3 分钟）的 mtime 窗口判断新文件：

```
CC 写卡 → mtime = T
深挖继续分析 5-8 分钟
CC 进程结束 → 检查 mtime → T 距今 5-8 分钟 >> 180 秒 → 漏检
```

深挖分析越长，漏检越严重。这是一种"用时间推测变化"的反模式——正确的做法是用快照记录"之前有什么"。

### 第一版修复：快照 diff（仍不够）

改成了 CC 启动前拍文件快照 → 结束后 diff 找新文件。原理正确，但仍有遗漏——只扫描了 `1-原子笔记/实体/` 和 `1-原子笔记/概念/` 两个硬编码子目录。CC 的 Write 工具可能写到任何已存在的子目录，包括 `报告输出/`——这也在扫描范围之外。

### 第二版修复：递归全量扫描（最终方案）

**scanExistingCards()** — CC 启动前递归遍历 `1-原子笔记/` 全部子目录，拍文件路径快照。

**verifyCardsWritten()** — CC 结束后递归遍历 `1-原子笔记/` 和 `报告输出/`，与快照做 diff。只有快照里没有的文件才算新文件。不依赖 mtime。

同时移除"未生成卡片"的粗暴提示——检测到就列路径，没检测到就安静。

### 卡片检测经验原则

| # | 原则 | 说明 |
|---|------|------|
| **8** | **diff 优于时间窗口** | "之前有哪些→现在有哪些"比"过去N分钟内改了什么"可靠得多。时间窗口在任何长时间运行的任务中都会失效 |
| **9** | **扫描要递归，不假设目录结构** | 硬编码子目录列表一定会有遗漏。CC 的 Write 目标是 OB vault 内任何位置——扫描必须覆盖全 vault |
| **10** | **检测不到就安静，不弹假警告** | 向用户报告"未生成卡片"的前提是你真的能确定没有生成。如果不能确定，保持沉默 |
| **11** | **插件和规则分层修改** | CLA.md 管 CC 行为（回复格式/分析深度），ChatView.ts 管插件行为（卡片检测/ui）。一条规则管一层，改了规则不改插件，改了插件不改规则 |

## 八、以后修改的注意事项

1. **保持信号比**：分析师人格永远占最大篇幅。不要因为"加一个功能"就往里塞写卡规则。
2. **用正面示例，不用禁止清单**：每加一条禁止格式，配一条"好的回复长这样"。
3. **171行是参考目标**：不是绝对不能超，但超过200行就该问自己"这个规则真的需要在插件版里吗"。
4. **嵌建分离**：自生长/进化/健康指标是工程底座的事，不要出现在插件版CLAUDE.md里。
5. **测试方法**：每次改完用一家没分析过的公司深挖测试——看回复是"对话"还是"交付物"。
6. **改行为改 CLA.md，改检测改 ChatView**：CC 怎么说怎么分析 → CLA.md。插件怎么检测怎么渲染 → ChatView.ts。一层改一个，不交叉。
7. **检测用 diff，不猜时间**：任何"有没有新文件"的判断，拍快照比 mtime 可靠。不假设任务多长时间会完成。
8. **检测不到就闭嘴**：不要向用户报告一个你无法确认的"失败"。假警告比不提示更破坏信任。
